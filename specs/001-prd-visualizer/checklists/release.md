# Release Gate — Acceptance & Doc-Fetch Requirements Quality Checklist

**Purpose**: "Unit tests for English" — validate that the acceptance scenarios, success criteria, and process-gate requirements (FR-018 doc-fetch, MVP scope, out-of-scope list) are well-written. Items are answerable YES/NO by reading the documents alone; this is NOT the runbook that walks the 7 acceptance scenarios.
**Created**: 2026-05-07
**Feature**: [spec.md](../spec.md)
**Scope**: 11 items, ~5 min walkthrough
**Audience**: spec author (pre-PR self-review)

---

## Acceptance scenario quality

- [ ] **CHK001** Each user story (US1–US5) has at least one acceptance scenario in Given/When/Then form, and each is independently testable per the spec's "Independent Test" section? — *Completeness* (spec.md User Stories)
- [ ] **CHK002** Are acceptance scenarios stated without referencing implementation details (no React/Express/specific port numbers leaked into user-facing scenarios)? — *Technology-agnostic check*
- [ ] **CHK003** Does each scenario specify a measurable threshold (timing, count, percentage), or rely on vague verbs like "quickly", "smoothly", "responsive"? — *Measurability*

## Success criteria quality

- [ ] **CHK004** Each SC-001 through SC-008 contains at least one numeric metric (time, %, count) — or is explicitly framed as a binary verifiable check? — *Measurability* (spec.md Success Criteria)
- [ ] **CHK005** Are success criteria stated in user/business outcome terms, or do they leak implementation language ("API returns 200", "React renders fast")? — *Technology-agnostic check*
- [ ] **CHK006** Is SC-008's "<35 minutes total" budget stated per-scenario (5 min × 7) or as a total ceiling — and is the chosen interpretation unambiguous? — *Clarity*

## Doc-fetch gate (FR-018)

- [ ] **CHK007** Does FR-018 specify *what* must be cited (Context7 ID or canonical doc URL) AND *where* (commit message), or only describe the duty in general terms? — *Clarity*
- [ ] **CHK008** Are the trigger conditions for FR-018 enumerated (new dep / new VSCode-API / library upgrade), or left as "any new library use"? — *Coverage*

## Scope boundaries

- [ ] **CHK009** Is the MVP scope unambiguously defined (which user stories are required for v0.1)? — *Clarity* (plan.md Implementation Strategy section, tasks.md Phase 3+4 = MVP)
- [ ] **CHK010** Are Phase-2 deferred items (drag-drop write-back, virtual scroll, status-bar badge, etc.) enumerated as a closed list in spec.md "Out of Scope", or left as open-ended "maybe later"? — *Completeness*

## Tagging criterion

- [ ] **CHK011** Is the v0.1.0 tagging criterion stated in terms of which scenarios + criteria must pass — and is "all 7 acceptance scenarios + SC-005 + SC-008" recoverable from spec text alone? — *Completeness*
